## 3. v0.3 의 정체성 통합 — '담음' 의 의미

### 3.1 평화의 두 함의 (결정 #45)

> "세움세상의 평화를 위한 그릇" — 사용자 직접 정의

**평화 (i) 기존 시스템과의 호환**
- `.docx` ↔ `.담음` 양방향 변환 가능 → 외부 도구와 공존
- 호스트 OS 위에서 세움 런타임이 .담음 읽고 실행
- 베어메탈 호환 *설계* (v0.5 에서 베어메탈 가능)

**평화 (ii) 한글 콘텐츠의 정확한 표현·압축**
- 자모 Huffman: 한국어 텍스트 압축률 ↑ (영어 zlib 의 일반 압축이 한국어에 약함)
- 형태소 chunk: 검색·의미 보존·번역 시 *의미 단위* 그대로
- UTF-8 +α: 자모 분리 형태도 binary 1급 표현
- 기존 포맷의 한글 무지를 *해소* — 이게 담음의 *raison d'être*

### 3.2 그릇 ≡ 담음 (결정 #46) — *명세 v1.5 의 가장 큰 통합*

> "나의세상은 세움으로 만들어진 담음의 형태 (그릇=담음)"

**v0.2 까지의 분리:**
| 추상 | v0.2 표현 |
|---|---|
| 그릇 (모듈) | C++ 함수 `register_time(env)` |
| 사용자 코드 | `.세움` 텍스트 |
| 환경 | 메모리 안 `Environment` 객체 |

**v0.3+ 의 통합:**
| 추상 | v0.3 표현 |
|---|---|
| 그릇 | `<이름>.담음` |
| 사용자 코드 | `사용자.담음` (`.세움` 컴파일 산출물) |
| 나의세상 인스턴스 | `세상.담음` (image 직렬화 결과) |

**모두 동일 binary 구조**. Smalltalk image + Lisp homoiconicity + WASM module 의 *한국어 통합*. **세움 세계에는 두 종류 객체가 없고 *하나만* — 담음**.

이는 *근본적 명세 변경*. v0.2 까지의 "그릇 = 도메인 기능 단위" (Ubiquitous Language v1.0 §3) 정의를 **"그릇 = 도메인 기능을 담은 .담음 파일"** 로 재정의.

### 3.3 builtin 그릇 → .담음 export 호환 (결정 #46 의 자기-검증)

v0.2 의 `시간` 그릇은 C++ 구현 (`src/modules/time.cc`). v0.3 ship 까지 이를 *.담음 export 가능* 으로 정합화:

1. v0.3 = `시간.담음 export` 함수 추가 — 호스트 C++ 환경에서 `시간` 그릇을 .담음 binary 로 출력
2. v0.3 = `.담음 import` 가 *.담음 형태의* `시간` 그릇을 등록 가능 (실제 사용은 builtin 우선)
3. v1.0 자기-호스팅 시 = 모든 그릇이 .담음 (C++ 의존 0)

→ 결정 #46 가 *자기-호스팅 길의 검증 지점*. v0.3 ship 후 사용자 그릇 정의 (.담음 export) 가 builtin 과 *동등* 함을 증명.

## 4. 담음 binary 포맷 v0.3 — Layout 초안

### 4.1 헤더 (16 bytes, fixed)

```
offset  size  field
0       4     magic       = "SEUM" (53 45 55 4D) ASCII (F1 patch)
                            *Why ASCII*: 4 bytes 고정 alignment, hex dump 에서 즉시 인식.
                            한국어 정체성은 chunk type 자모 (§5) 와 한글 텍스트 chunk 내부에 보존.
4       2     version     = 0x0001 (v0.3)
6       2     flags       = bit0: 압축됨(자모Huffman), bit1: 베어메탈호환,
                            bit2: 형태소포함, bit3: 외부.담음참조있음
8       4     chunk_count = u32 LE
12      4     header_crc  = u32 LE (CRC32 over bytes 0..11)
```

**Alignment**: 헤더 끝 = 16-byte 경계. 모든 chunk 시작도 16-byte aligned (베어메탈 부트로더 친화).

### 4.2 Chunk Table (각 chunk 16 bytes)

```
offset  size  field
0       1     chunk_type  = 1 byte (§5 자모 매핑)
1       1     compression = 0 = raw, 1 = zlib (옵셔널), 2 = 자모Huffman (한글텍스트)
2       2     flags       = chunk 별
4       4     offset      = u32 LE — 본문 데이터 시작 offset
8       4     size        = u32 LE — 데이터 크기 (압축 후)
12      4     uncompressed_size = u32 LE — 압축 전 (raw 면 size 와 같음)
```

Chunk Table 은 헤더 직후 16 * chunk_count bytes.

### 4.3 Chunk Body (alignment: 16 bytes 패딩)

각 chunk 본문은 16-byte boundary 시작. 본문 크기 < align → 0x00 패딩.

### 4.4 Inline 필수 5 chunks 의 binary 형태

| chunk_type | 자모 식별 | 본문 binary |
|---|---|---|
| `코드` | `ㄱ` | bytecode::Program 직렬화 (functions 별 chunk + main) |
| `이름` | `ㄴ` | 그릇 이름·param 시그니처·export 목록 (UTF-8 길이접두) |
| `메타` | `ㄷ` | 작성자/시각(SST)/세움 버전/체크섬/의존성 목록 |
| `상태` | `ㅁ` | 정적 상수 풀 (CONST_INT/STR pool) + 나의세상 인스턴스면 변수 binding |
| `한글텍스트` | `ㅎ` | 자모 Huffman 압축 본문 + 형태소 토큰 인덱스 |

### 4.5 옵셔널 3 chunks

| chunk_type | 자모 식별 | 본문 |
|---|---|---|
| `미디어` | `ㅂ` | raw bytes + MIME-like 작은 매니페스트 (v0.3 codec X) |
| `소스` | `ㅅ` | `.세움` 원본 텍스트 |
| `링크` | `ㅇ` | 외부 .담음 파일 경로 (상대 또는 같은 시스템 안 절대) |

**총 8 chunk types**. 사용 가능 한글 자모: ㄱ/ㄴ/ㄷ/ㄹ/ㅁ/ㅂ/ㅅ/ㅇ/ㅈ/ㅊ/ㅋ/ㅌ/ㅍ/ㅎ (14 자음). 8개 매핑 + 6개 예약 (v0.4+).

### 4.6 베어메탈 호환 설계

포맷이 *부트로더가 직접 로드 가능* 하도록:
- 헤더 magic = identifier 인식 가능
- 16-byte alignment = 일반 메모리 페이지 (4KB) 의 약수
- 코드 chunk = bytecode::Function 들이 위치 독립적 (relative jumps only — v0.2 의 4B 절대 offset 은 chunk 내 한정)
- 상태 chunk = 초기 메모리 이미지 (load address 가정 시 메모리 mapped 가능)
- 메타 chunk = entry point 위치 (main Function 의 offset)

**v0.5 에서 추가될** (현재 미구현): 부트로더 작성 + 메모리 매니징 + 드라이버. v0.3 의 *포맷 설계* 가 v0.5 진입 시 *재설계 0* 임을 약속.

## 6. 양방향 변환 (결정 #47) — `.txt ↔ .담음`

### 6.1 import (`.txt → .담음`)

```
세움 변환 hello.txt -> hello.담음
```

- 파일 내용을 UTF-8 로 읽음
- 한글텍스트 chunk (ㅎ) 에 자모 Huffman 압축으로 저장
- **형태소 토큰화** (F6 patch): v0.3 = 공백·마침표·쉼표·괄호 기준 단순 분리만. *진정한 형태소 분석기* (조사 분리, 어간 추출) 는 v0.4+ 백로그. 명세는 "형태소" 단어를 *호환적 미래보기* 용어로 사용.
- 메타: 원본 파일명, 변환 시각, 원본 인코딩

### 6.2 export (`.담음 → .txt`)

```
세움 변환 hello.담음 -> hello.txt
```

- 한글텍스트 chunk 디코드 → UTF-8
- 메타에 원본 파일명 있으면 비교 (round-trip 검증)
- *비파괴*: import-export 후 비트 단위 동일

### 6.3 round-trip 보존 (결정 #52 의 c)

`.txt → .담음 → .txt` 가 *비트 단위 동일* 해야 함. 자모 Huffman 의 *손실 없는* 검증.

## 10. Module Design — v0.3 Deep Module 스케치

### 10.1 `src/dameum.{h,cc}` — 담음 reader/writer

**공개 인터페이스 (4 심볼 목표)**:
```cpp
namespace seum::dameum {
    struct Container;
    Container read (const std::string& utf8_path);     // 디스크 → 메모리
    void     write(const Container& c, const std::string& utf8_path);
    std::u32string disassemble(const Container& c);   // 한글 dump (디버그용)
}
```

**Container 데이터:**
```cpp
struct Container {
    std::uint16_t version;
    std::uint16_t flags;
    std::vector<Chunk> chunks;
    // Chunk = { type (자모), compression, raw_bytes, ... }
};
```

**왜 Deep**: 외부는 4 심볼만 본다. binary parsing, Huffman, 형태소 토큰, alignment, CRC, magic — 전부 내부.

### 10.2 `src/jamo_huffman.{h,cc}` — 한국어 자모 압축

**공개 (2 심볼)**:
```cpp
namespace seum::jamo {
    std::vector<std::uint8_t> compress  (const std::u32string& utf32);
    std::u32string            decompress(const std::vector<std::uint8_t>& compressed);
}
```

내부: 자모 빈도 table, Huffman tree 생성, bit-level encode/decode.

### 10.3 `src/morpheme.{h,cc}` — 형태소 분리 (v0.3 간단)

**공개 (1 심볼)**:
```cpp
namespace seum::morpheme {
    std::vector<std::u32string> tokenize(const std::u32string& text);  // v0.3 = 공백+마침표
}
```

v0.4+ 에서 본격 형태소 분석기로 확장 가능.

### 10.4 호출 그래프

```
세움 빌드 hello.세움:
  parse → ir::lower → bytecode::compile → dameum::write(Container{코드,이름,메타,...})

세움 실행 hello.담음:
  dameum::read → bytecode::Program 재구성 → vm::run

세움 변환 hello.txt -> hello.담음:
  read text → jamo::compress + morpheme::tokenize → dameum::write(Container{한글텍스트})
```
