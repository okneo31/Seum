## 4. 점검 산출 추가 잠긴 결정 (79–87)

|---|---|

**타입 매개변수 (1G)**: `이름: 타입` 형태. v0.4 = 자산 타입(BTC·KRW) 한정으로 도입. 일반 타입 시스템(정수·문자열 매개변수 타입 주석)은 v0.5+ 백로그. 무타입 매개변수(`함수 더하기(a, b)`)는 그대로 유효.

**누적 한글 키워드 (v0.4 신규: 계약·자산·받아서·누르면 활성 + 게임세상·화면구성 등 그릇 키워드)** — 그릇 키워드는 그릇 로드 시 등록되는 *문맥 키워드*로 처리 가능 (코어 예약어 최소화).

## 8. 담음 포맷 v0.4 확장 (version 0x0002, 결정 86)

```
헤더 flags:  bit0 압축 / bit1 베어메탈 / bit2 형태소 / bit3 외부참조 / bit4 암호화chunk포함(신규)
chunk flags: 암호화 비트 추가 — 암호화는 compression과 직교 (압축 후 암호화)
chunk type:  ㅈ = 자산 장부 (신규) — 계약 자산 캐시 스냅샷, byte 값 0x09
```

| 자모 | chunk | byte | 상태 |
|---|---|---|---|
| ㄱ~ㅎ (8종) | 코드·이름·메타·상태·미디어·소스·링크·한글텍스트 | 0x01~0x08 | v0.3 |
| **ㅈ** | **자산 장부** | **0x09** | **v0.4 신규** |
| ㄹ·ㅊ·ㅋ·ㅌ·ㅍ | — | — | 예약 (v0.5+) |

reader는 version 분기 — `0x0001`(v0.3) 하위호환, `0x0002`(v0.4) 신규 필드 인식.

## 10. Module Design — v0.4 Deep Module 스케치

### 10.1 native bridge 분할 (점검 2G)

현재 `native_bridge.{h,cc}`는 native 2개를 단일 `register_natives`로 등록. v0.4는 ~20개 추가 → 도메인별 분할로 Deep 유지:

```
src/modules/native_bridge.h        — 공통 진입점
src/modules/native_금융.{h,cc}     — TLS·BTC RPC·암호화·키스토어
src/modules/native_gui.{h,cc}      — 호스트 OS 창·렌더·이벤트 루프 (게임·웹 공유)
src/modules/native_http.{h,cc}     — HTTP 클라이언트·서버
```

각 모듈 공개 인터페이스는 `register_xxx(Environment&)` 1심볼. 내부 = 호스트 OS API 전부 은닉.

### 10.2 계약 컨테이너 (점검 1K, 결정 87)

`bytecode::Program` 확장 — 계약은 `{ 자산 필드 목록, 비-자산 상태 슬롯, 메서드 함수 목록 }` 섹션. .담음 직렬화 시 메서드 = ㄱchunk, 비-자산 상태 = ㅁchunk, 자산 캐시 = ㅈchunk.

### 10.3 이벤트 루프 (결정 82·84)

단일 `EventLoop` 모듈 — `--vm`/`--tree` 양 경로가 공유. 콜백 = 1급 함수값(80), 디스패치 시 호출 측 엔진으로 실행.

## 12. v1.7.1 패치 — 점검 3·4회차 산출 (2026-05-23)

> v1.7 발행 후 점검 2회 추가 실시 — **3회차** 원본 예시 손-파싱 / **4회차** 적대적 엣지케이스. 서로 다른 방법이 서로 다른 종류의 공백을 포착.
> 집계: CRITICAL **0** / HIGH 4 / MEDIUM 9 / LOW 2. 새 CRITICAL 0 → v1.7 핵심 견고. HIGH 4 + MEDIUM/LOW = 결정 88–92로 흡수.
> HIGH 4건은 진짜 설계 갈림길이라 옵션 선택으로 잠금 — 4건 전부 권고안 채택.

### 12.1 HIGH 해소 — 잠긴 결정 88–91

| # | 결정 |
|---|---|
| 88 | **종결자 규칙** (3-1 해소). 선언형 블록 `{ }` *내부* item은 종결자 `.` 생략 — 줄바꿈이 item 구분. 그 외 모든 문장은 결정 #10대로 `.` 필수. 원본 `프로그래밍예시.md` 3종은 이 규칙으로 그대로 작동. v0.0~v0.3 코드(전부 `.` 사용)는 무영향 |
| 89 | **범용 후행 블록 호출** (3-2 해소). 임의 호출식에 `f(인자) { 블록 }` 후행 블록 허용 (Kotlin trailing lambda류). 블록 = 콜백 등록부. `ast.h::CallExpr`에 옵셔널 `trailing_block` (vector<Stmt>) 추가. `버튼("라벨") { 누르면 -> … }`가 1급 문법 |
| 90 | **선언형 블록 멤버 = 전역 등록** (4-1 해소). `계약`·`게임세상`·`화면구성` 안의 명명 멤버(`캐릭터 용사` 등)는 전역 이름으로 등록. `누르면` 콜백은 멤버를 *전역 조회* — closure 캡처 불필요. §11의 "클로저 v0.5+"를 *강화*. 결정 #34(글로벌 읽기 OK)와 정합. 원본 예시 콜백 전부 이 방식으로 작동 |
| 91 | **이벤트 루프 종료 조건** (4-2 해소). 이벤트 루프는 *활성 이벤트 소스*(열린 창·listen 중 서버·등록 타이머)가 0개가 되면 종료. main 코드가 끝나도 활성 소스가 있으면 루프 계속. 창 전부 닫힘·서버 전부 중지 = 자연 종료. Node.js·GUI 앱 표준 모델. 결정 82 보강 |

### 12.2 MEDIUM·LOW 해소 — 잠긴 결정 92

결정 92 = v1.7.1 점검 정정 묶음. 아래 11개 소(小)정정 일괄:

| finding | 등급 | 정정 |
|---|---|---|
| 3-3 | MEDIUM | `함수 … 받아서 -> X { }` — `받아서` 있으면 `-> X`의 X는 *누적 대상*, 없으면 *결과명*. `FuncDeclStmt`에 `accumulate_target` 필드 추가, 파서는 `받아서` 키워드 유무로 분기 |

### 12.3 점검 4종 방법의 상보성

| 점검 | 방법 | 포착한 공백 종류 |
|---|---|---|
| 1회차 | 명세 내부 정합성 | 결정 간 *모순* (예: `.` 충돌) |
| 2회차 | 실제 코드 대조 | 명세 ↔ *코드 현실* 격차 (예: `+=` 토큰 부재) |
| 3회차 | 원본 예시 손-파싱 | *문법* 공백 (call-with-block, 종결자) — 예시를 파싱해야만 보임 |
| 4회차 | 적대적 엣지케이스 | *의미·런타임* 공백 (closure, 루프 종료) — "깨는 입력"을 던져야 보임 |

1·2회차는 *결정이 서로/코드와 어긋남*을, 3·4회차는 *결정이 닿지 않은 영역*을 잡음. 4종이 상보적 — 어느 하나만으로는 불충분.
